## 1. 기술 면접 대비 (Q&A)

### Q1. CSS Flexbox 레이아웃 내에서 텍스트 말줄임표(truncate)가 적용된 자식 요소 때문에 레이아웃이 깨지거나 다른 자식 요소가 밀리는 현상이 발생했을 때, 어떻게 해결했나요?

**A1. 문제 인식 ➔ 대안 분석 및 Trade-off 평가 ➔ 최종 설계 합리화**

* **문제 인식**:
  상세보기 모달에서 식당 명칭이 길어질 경우, 우측에 위치한 '룰렛 추가/제외' 버튼이 모달 영역 밖으로 밀려나 레이아웃이 깨지는 현상이 발생했습니다. 식당 명칭을 렌더링하는 `<DialogTitle>` 엘리먼트에 `truncate` (`overflow: hidden; text-overflow: ellipsis; white-space: nowrap;`) 클래스를 부여했음에도 텍스트가 잘리지 않고 버튼을 밀어냈습니다. 이는 CSS Flexbox 표준 스펙상 flex item의 `min-width` 기본값이 `auto`(또는 `min-content`)로 지정되어 있어, 텍스트가 줄어들지 않고 고유의 너비를 강제하려 하기 때문입니다.

* **대안 분석 및 Trade-off 평가**:
  * **대안 1: 모달 팝업의 너비(`max-w-md` ➔ `max-w-lg`)만 단순히 키우기**
    * *장점*: 공간이 넓어져 보다 많은 텍스트를 한눈에 볼 수 있습니다.
    * *단점*: 모바일 환경(예: 375px 이하 화면)에서는 팝업의 절대 너비가 결국 뷰포트 너비로 제한되므로 동일한 레이아웃 깨짐 현상이 여전히 발생합니다.
  * **대안 2: flex item인 `<DialogTitle>`에 `min-w-0` 적용**
    * *장점*: 화면 크기와 관계없이 flex item이 최소 콘텐츠 크기 이하로 축소(`min-width: 0`)될 수 있게 하므로 `truncate`가 정상적으로 동작하여 레이아웃이 안전하게 유지됩니다.
    * *단점*: 식당 이름이 극단적으로 길면 뒷부분이 가려져 생략되는 한계가 있습니다.

* **최종 설계 합리화 (이중 대응)**:
  사용자 경험을 극대화하기 위해 **대안 1과 대안 2를 조합한 이중 대응 설계**를 채택했습니다. 
  1. 모달의 최대 너비를 기존 `max-w-md`(448px)에서 `max-w-lg`(512px)로 변경하여 데스크톱 및 태블릿 화면에서 시원하고 쾌적한 뷰를 제공합니다.
  2. 동시에 `<DialogTitle>`에 `min-w-0` 클래스를 부여하여 모바일 기기 또는 극단적으로 긴 텍스트 입력 시에도 레이아웃이 유연하게 축소되면서 말줄임표(`...`) 처리되도록 방어막을 구축했습니다. 이를 통해 어떤 디바이스 환경에서도 깨짐이 발생하지 않는 반응형 UI를 달성했습니다.

### Q2. Tailwind CSS와 `tailwind-merge` (`cn` 유틸리티) 사용 중, 커스텀 컴포넌트 내부의 반응형 유틸리티 클래스(예: `sm:max-w-sm`)가 외부에서 주입한 기본 클래스(예: `max-w-lg`)보다 우선적으로 적용되어 스타일 오버라이딩이 되지 않는 문제를 어떻게 해결했나요?

**A2. 문제 인식 ➔ 대안 분석 및 Trade-off 평가 ➔ 최종 설계 합리화**

* **문제 인식**:
  공통 모달 컴포넌트인 `DialogContent`에는 기본적으로 `sm:max-w-sm`이 정의되어 있었습니다. `RestaurantDetailModal`에서 이를 사용할 때 너비를 늘리기 위해 `max-w-lg`를 전달했으나, 데스크톱 화면에서 모달 너비가 `sm:max-w-sm`(384px)로 고정되어 넓어지지 않는 문제가 발생했습니다.
  `tailwind-merge`는 접두사가 다른 클래스(예: 접두사가 없는 `max-w-lg`와 `sm:` 접두사가 있는 `sm:max-w-sm`)를 서로 다른 스타일 그룹으로 인식하여 병합하지 않고 둘 다 클래스에 남겨둡니다. 그 결과 CSS 명시성 규칙에 의해 `@media (min-width: 640px)` 조건 하에 있는 `sm:max-w-sm`이 항상 우선 적용됩니다.

* **대안 분석 및 Trade-off 평가**:
  * **대안 1: 공통 컴포넌트(`dialog.tsx`)의 `sm:max-w-sm` 제거**: 공통 컴포넌트의 기본 레이아웃을 훼손하여 다른 페이지의 모든 모달 크기에 영향을 주므로 부적합합니다.
  * **대안 2: `!important` 접두사 사용 (`!max-w-lg`)**: 스타일 우선순위를 강제로 끌어올릴 수 있으나 Tailwind CSS의 유지보수성을 떨어뜨리고 반응형 대응(모바일에서의 100% 너비 등)을 방해합니다.
  * **대안 3: 외부 주입 클래스에 동일한 반응형 접두사 매칭 (`sm:max-w-lg`)**: `tailwind-merge`가 동일한 `sm:` 접두사를 확인하고 `sm:max-w-sm`을 `sm:max-w-lg`로 안전하게 덮어쓰도록 유도합니다.

* **최종 설계 합리화**:
  **대안 3**을 선택하여 모달을 호출할 때 `sm:max-w-lg`를 명시적으로 주입했습니다. 이로 인해 `tailwind-merge`에 의해 공통 컴포넌트의 `sm:max-w-sm`이 제거되고 `sm:max-w-lg`로 치환되며, 모바일 환경(`max-w-md`)과 데스크톱 환경(`max-w-lg`) 모두 최적의 크기로 정확히 렌더링되는 깔끔한 오버라이딩을 달성했습니다.

### Q3. Flexbox 레이아웃 중첩 시 자식의 `truncate`가 동작하지 않고 부모 컨테이너가 늘어나는 현상이 발생하는 이유와 해결 방법은 무엇인가요?

**A3. 문제 인식 ➔ 대안 분석 및 Trade-off 평가 ➔ 최종 설계 합리화**

* **문제 인식**:
  부모 엘리먼트(`flex flex-col`) 안에 가로 배치를 위한 자식 엘리먼트(`flex items-center justify-between`)가 있고, 그 내부의 `<DialogTitle>`에 `min-w-0`과 `truncate`를 지정했음에도 말줄임이 되지 않고 버튼을 밀어내는 현상이 나타났습니다.
  이는 중간에 있는 가로 배치 `div` 역시 세로 `flex` 컨테이너의 자식(flex item)이므로, 명시적인 너비 제한이 없으면 텍스트의 실제 너비에 맞춰 무한히 확장되기 때문입니다.

* **해결 방법**:
  중간 레이어의 가로 배치 flex container인 `div`에 `w-full`과 `min-w-0`을 추가하여 너비가 부모 컨테이너의 영역을 초과하지 않도록 제한했습니다. 이로써 너비 제한 제약이 맨 하위 자식인 `<DialogTitle>`까지 올바르게 전파되어 정상적으로 `truncate`와 `flex-1`을 통한 말줄임 처리가 동작하게 되었습니다.

### Q4. 실시간 다인원 룰렛 추천 및 맛집 관리 서비스(식권대장)의 사용자 트래픽 분석과 페이지 성능 모니터링을 위해 Vercel Analytics를 도입한 배경과, 그 설계적/비용적 고려사항은 무엇인가요?

**A4. 문제 인식 ➔ 대안 분석 및 Trade-off 평가 ➔ 최종 설계 합리화**

* **문제 인식**:
  실시간 다인원 룰렛 추천 서비스의 이용 현황(사용자가 어떤 맛집 카드를 선호하는지, 룰렛을 얼마나 자주 돌리는지, 협업 방을 얼마나 생성하는지 등)과 페이지별 지연 시간을 추적하여 UX를 개선해야 합니다. 하지만 클라이언트가 가벼워야 하는 대시보드 성격의 웹 앱에서 무겁고 복잡한 분석 스크립트를 탑재하는 것은 초기 페이지 로딩(LCP)과 성능을 해칠 우려가 있었습니다.

* **대안 분석 및 Trade-off 평가**:
  * **대안 1: Google Analytics (GA4) 도입**
    - *장점*: 풍부한 마케팅 분석 도구와 유저 코호트 분석이 가능함.
    - *단점*: 무거운 번들 크기(약 80KB+)로 인해 초기 로딩 지연이 발생할 수 있고, 다국어 및 쿠키 동의서 팝업 설치가 까다로우며 무상 상태 유지가 어려움.
  * **대안 2: Vercel Web Analytics 도입 (채택)**
    - *장점*: 단 1KB 미만의 극도로 가벼운 스크립트만 클라이언트에 서빙됨. Next.js App Router의 `layout.tsx` 내에 `<Analytics />` 컴포넌트 마운트 한 줄로 연동이 끝남. 개인정보 보호 규정(GDPR)을 완벽히 충족하여 성가신 쿠키 동의 팝업이 전혀 필요 없고, Vercel 배포 시 완전한 제로 구성(Zero-Config)으로 연동됨.
    - *단점*: Vercel 호스팅 인프라에 종속됨.

* **최종 설계 합리화**:
  웹 서비스의 초기 진입 속도를 극대화하고, 복잡한 비즈니스 분석보다는 간단하고 정확한 실시간 방문자 행동(Page View, Referrer, 브라우저 환경) 및 Web Vitals 성능 지표(LCP, FID, CLS) 모니터링이 핵심 목적이므로 **Vercel Web Analytics(대안 2)**를 채택하여 구현했습니다. 루트 `layout.tsx`에 컴포넌트를 배치함으로써 번들 오버헤드를 최소화하고 안전하게 실시간 사용자 데이터를 수집할 수 있는 기반을 구축했습니다.

### Q5. 네이버 지도 크롤링 시, 데이터 소스(음식점 상호명)의 특수문자(괄호) 및 위치 기반 지역명 접미사 중복 등으로 인해 매칭률이 떨어지는 문제를 어떻게 해결하셨나요?

**A5. 문제 인식 ➔ 대안 분석 및 Trade-off 평가 ➔ 최종 설계 합리화**

* **문제 인식**:
  `sik_list.json`에 있는 음식점 데이터 중 `아비꼬(가락시장역점)`와 같이 괄호`()` 및 내부 지점명이 혼합된 명칭이 다수 존재했습니다. 기존 크롤러는 위경도 좌표를 기반으로 기계적인 지역 접미사(예: `" 가락동"`)를 붙여 `"아비꼬(가락시장역점) 가락동"` 과 같은 쿼리를 구성하여 네이버 검색을 요청했습니다. 이로 인해 검색 엔진의 매칭 오류(지점명과 지역명의 중복, 특수문자에 의한 노이즈 등)가 발생해 84개 매장 중 22개 매장(약 26%)의 Place ID 및 메뉴 수집이 실패하는 현상이 발생했습니다.

* **대안 분석 및 Trade-off 평가**:
  * **대안 1: 기계적 접미사 전면 비활성화 및 음식점 고유명으로만 검색**
    * *장점*: 상호명에 이미 지점명이 표기된 프랜차이즈의 경우 매칭 확률이 높아집니다.
    * *단점*: `"김가네"`, `"예원"` 등 전국에 수많은 지점이 있는 동명의 음식점들은 가락동 좌표 근처가 아닌 엉뚱한 타지역 지점의 정보가 크롤링될 위험(오매칭)이 매우 높습니다.
  * **대안 2: 정규식 기반 쿼리 전처리 필터링 도입 (채택)**
    * *장점*: 괄호 `(...)`를 공백으로 치환하여 자연스러운 텍스트(`아비꼬 가락시장역점`)로 정제합니다. 또한, 정제된 이름에 이미 지역 접미사(예: `"가락"`)가 포함된 경우 중복 접미사 조립을 생략하여 검색 정확도를 극대화합니다.
    * *단점*: 정규식 패턴 연산 및 쿼리 파싱을 위한 오버헤드가 발생하지만, 서버 사이드가 아닌 수집 스크립트(1회성/정적 배치)이므로 런타임 성능에 전혀 지장이 없습니다.

* **최종 설계 합리화**:
  데이터의 무결성과 오매칭 방지를 동시에 해결하기 위해 **대안 2(쿼리 전처리 필터링)**를 적용했습니다. 괄호를 공백으로 치환하는 정규식을 적용하고, `area` 접미사를 덧붙일 때 상호명에 이미 그 접미사의 키워드(예: `가락`, `대학`, `청량`, `공덕`)가 포함되어 있다면 접미사 덧붙이기를 생략하도록 쿼리 조립 로직을 개선했습니다. 
  개선 후, Place ID를 찾지 못했던 14개 매장 중 아비꼬(가락시장역점)을 포함한 7개 매장의 데이터(Place ID 및 세부 메뉴 정보)를 정상적으로 추가 수집(매칭 실패율 50% 개선) 하는 데 성공했습니다.

### Q6. 카카오 지도 SDK는 다크모드 타일을 공식적으로 지원하지 않습니다. 서비스의 어두운 다크 테마(예: `#0f1015`)와 지도의 밝은 백색 타일이 충돌하는 문제를 어떻게 해결했고, 이 과정에서 커스텀 HTML 마커의 색상 왜곡을 어떻게 방지했나요?

**A6. 문제 인식 ➔ 대안 분석 및 Trade-off 평가 ➔ 최종 설계 합리화**

* **문제 인식**:
  다크모드 활성화 시 카카오 지도는 기본 백색 지도를 서빙하여 대시보드의 네이비 톤 어두운 배경과 시각적 조화를 해쳤습니다. 단순 전체 필터를 걸면 지도 타일은 다크해지지만, 그 위에 뜨는 커스텀 HTML 마커와 상세 카드 오버레이까지 역필터링되어 마커 색상 정보(예: 카테고리별 테마색)가 변질되는 치명적인 문제가 있었습니다.

* **대안 분석 및 Trade-off 평가**:
  * **대안 1: Mapbox 등 다크 맵을 기본 지원하는 상용 맵으로 전면 개편**
    - *장점*: 코드가 깔끔해지고 SDK 레벨에서 다크 테마를 공식 서빙받을 수 있습니다.
    - *단점*: 한국 내 상세 번지수나 대중교통 인프라 매핑 정확도가 카카오/네이버에 비해 낮고, 유료 비용 플랜 전환 및 API Key 연동 코드를 모두 새로 작성해야 하는 큰 공수가 듭니다.
  * **대안 2: 정밀 CSS 선택자 기반의 부분적 CSS Filter 역반전 처리 (채택)**
    - *장점*: 카카오 지도 SDK를 그대로 유지하면서, 지도 내 배경 이미지 레이어(`.kakao-map-container img`)만 선택해 `invert` 및 `hue-rotate` 필터를 걸어 다크모드를 완벽하게 구현할 수 있습니다. HTML `div`로 그린 커스텀 마커와 카드 컴포넌트는 이미지 필터링 대상에서 배제되므로 고유 색상을 온전히 유지합니다.
    - *단점*: 지도 타일의 렌더링 구조(클래스명, 태그)가 SDK 업데이트 시 변경되면 깨질 수 있으나, 카카오 지도의 경우 배경 타일은 표준 `img` 태그로만 렌더링되므로 안정성이 높습니다.

* **최종 설계 합리화**:
  비용 절감과 구현 신속성, 사용자 가독성 보존을 모두 챙기기 위해 **대안 2**를 채택했습니다. `.dark .kakao-map-container img` 선택자를 정의하여 배경 타일 이미지만 반전(`invert(92%) hue-rotate(180deg) brightness(95%) contrast(90%)`)시켜 세련된 네이비 다크 맵을 형성했고, 텍스트와 색상이 온전히 보존되어야 하는 HTML 오버레이는 그대로 가독성을 지키도록 설계하여 비용 제로의 고품질 하이브리드 다크맵을 렌더링하는 데 성공했습니다.

### Q7. 무작위 룰렛 추천 기능 구현 시, 단순 그라디언트 원판 이미지만 회전시키는 방식(AI 생성물 특유의 슬롭 디자인)을 탈피하고, 실제 룰렛 후보 식당들을 시각화하며 회전 바늘이 가리키는 파이 조각과 실제 당첨 결과 데이터가 정확히 일치하도록 휠 물리를 어떻게 설계했나요?

**A7. 문제 인식 ➔ 대안 분석 및 Trade-off 평가 ➔ 최종 설계 합리화**

* **문제 인식**:
  기존 룰렛은 정체불명의 무지개색 그라디언트 판만 돌고 당첨 팝업이 뜨는 형태로, 'AI가 대충 만들어 놓은 장식용 컴포넌트'라는 인상을 주었습니다. 또한 룰렛 포인터 화살표가 가리키는 지점과 팝업의 실제 당첨 식당이 일치하지 않아 사용성 신뢰가 매우 낮았습니다.

* **대안 분석 및 Trade-off 평가**:
  * **대안 1: Canvas API 기반의 동적 룰렛 그리기 및 requestAnimationFrame 루프 제어**
    - *장점*: 픽셀 단위로 정교한 룰렛판 및 파티클을 묘사할 수 있습니다.
    - *단점*: 룰렛에 들어가는 텍스트와 React State(맛집 추가/제외 상태) 간의 동기화가 어렵고, 스크린 리더 등 웹 접근성에 취약합니다.
  * **대안 2: React State + SVG 동적 Sectors + 정밀 물리 회전 공식 연계 (채택)**
    - *장점*: 룰렛 후보 배열 길이에 맞춰 SVG `<path>` 및 `<text>` 요소가 즉각적으로 바인딩되며, CSS 하드웨어 가속을 받는 CSS Rotate와 Spring 물리 전환이 결합되어 반응 속도가 빠르고 쫀득한 마찰감을 제공합니다.
    - *단점*: SVG 패스 렌더링을 위해 각도와 원호(Arc) 계산 수식이 들어가야 합니다.

* **최종 설계 합리화**:
  React와 유기적으로 상태가 공유되며 고유한 인터랙션을 완성하기 위해 **대안 2**를 채택했습니다. 
  1. 원호 수식(`describeArc`)을 활용해 후보 개수 $K$에 비례한 SVG 슬라이스 패스를 생성하고, 각 슬라이스 중심선에 맞추어 식당 이름 라벨을 radial 형태로 회전 배치했습니다.
  2. 화살표가 상단(12시 방향, $0^\circ$ 혹은 $360^\circ$ 축)에 고정되어 있으므로 당첨된 인덱스 `randomIndex`가 회전 종료 후 12시 방향에 수직으로 정지할 수 있도록 회전각 물리 공식인 `targetRotation = currentRotationOffset + 5회전(1800deg) + (360 - sliceCenterAngle) % 360`을 적용했습니다. 
     * **트러블슈팅 디버그**: 기존 수식은 0도 기준선이 3시 방향에 있는 극좌표계를 기준으로 하여 바늘 위치를 $270^\circ$로 오프셋 설정했으나, `describeArc` 수식 내부에서 이미 12시 방향으로의 축 변환($-90^\circ$)을 수행하고 있어 $90^\circ$의 각도 불일치(샐러디 당첨 시 서울익스프레스 가리킴)가 발생하고 있었습니다. 기준 오프셋을 $360^\circ$로 보정하여 시각적 화살표 포인터가 당첨 식당 슬라이스의 정중앙을 정확하게 가리키며 쫀득하게 멈춰 서는 일관성 있는 UX를 달성했습니다.

### Q8. 당첨 축하 팝업(WinnerModal)처럼 다수의 그래픽 파티클이 동적으로 움직이는 화면에서 INP(Interaction to Next Paint) 수치를 최적화하고 프레임 드랍을 막기 위해 어떤 렌더링 기법을 도입하셨나요?

**A8. 문제 인식 ➔ 대안 분석 및 Trade-off 평가 ➔ 최종 설계 합리화**

* **문제 인식**:
  당첨 모달이 뜨는 순간 80여 개의 컨페티 입자들이 폭발하는 연출을 할 때, 각 입자의 속성(색상, 딜레이, 크기, Border Radius)을 리액트 map 루프 내부 인라인 스타일에서 `Math.random()` 호출로 바인딩하고 있었습니다. 이 경우 모달이나 입자 상태가 업데이트되어 리렌더링될 때마다 모든 입자의 Border Radius와 크기가 매 프레임 재계산되는 연산 낭비와 스타일 플리커링(flickering)이 유발되며 FPS가 저하되었습니다.

* **대안 분석 및 Trade-off 평가**:
  * **대안 1: Canvas로 컨페티 애니메이션 루프 그리기**
    - *장점*: DOM 노드가 늘어나지 않아 가상 DOM 오버헤드가 전혀 없습니다.
    - *단점*: 단지 축하 팝업 하나를 위해 캔버스를 항상 마운트해 두고 모니터링하는 복잡도가 증가합니다.
  * **대안 2: 고유 상태 분리(Stable Properties) 및 GPU 가속 키프레임 애니메이션 (채택)**
    - *장점*: 입자 생성 직후(useEffect 최초 1회) 모든 무작위 속성 값을 계산하여 React State 배열에 완전히 고정합니다. 렌더링 루프에서는 상태로 정의된 고유 값을 스타일로 참조하고, 애니메이션은 CPU 레이아웃 연산(Reflow)을 유발하지 않는 `transform`과 `opacity` 속성만을 이용해 Framer Motion keyframes로 실행하므로 60fps로 매끄럽게 동작합니다.

* **최종 설계 합리화**:
  React와 Framer Motion의 강점을 살려 성능과 코드 간결성을 모두 쥐기 위해 **대안 2**를 채택했습니다. 컨페티 데이터 구조에 `size`, `delay`, `borderRadius`, `rotateDir`을 명시해 상태에 고정시킴으로써 렌더링 당 시각적 잡음을 100% 제거하고, 브라우저가 레이아웃 재배치 과정 없이 GPU 가속을 통해 입자 궤적을 부드럽게 렌더링하도록 유도하여 INP 속도를 극적으로 개선했습니다.

### Q9. 데스크톱 브라우저 환경에서 카테고리 칩 선택 바와 같이 가로 스크롤이 긴 UI 요소를 마우스 드래그로 탐색하고 편리하게 클릭할 수 있는 수평 캐러셀 슬라이더를 구현하는 과정에서 발생한 클릭 이벤트 간섭 문제를 어떻게 해결하셨나요?

**A9. 문제 인식 ➔ 대안 분석 및 Trade-off 평가 ➔ 최종 설계 합리화**

* **문제 인식**:
  데스크톱 브라우저는 모바일 기기나 트랙패드와 달리 마우스만으로는 기본 수평 스크롤바 조작 외에 드래그 스크롤을 지원하지 않아 가로로 긴 카테고리 칩 리스트 탐색이 불편합니다. 이를 개선하기 위해 Mouse Down, Move, Up 이벤트를 감지해 마우스 드래그 거리에 비례하여 컨테이너를 수평 스크롤하는 드래그 스크롤을 구현했습니다. 하지만 마우스 드래그 동작이 마우스 Up 시점에 브라우저 기본 동작에 의해 자식 카테고리 버튼의 `click` 이벤트로 격발되어, 드래그 스크롤을 마쳤을 때 사용자가 의도치 않게 마지막 칩을 선택(오클릭)하는 심각한 사용성 간섭 버그가 발생했습니다.

* **대안 분석 및 Trade-off 평가**:
  * **대안 1: 드래그 중인 상태를 state로 관리하고 모든 자식 버튼의 `disabled` 처리하기**
    - *장점*: 확실하게 클릭을 막을 수 있습니다.
    - *단점*: 드래그가 시작되고 끝날 때마다 매번 리렌더링이 발생하여 성능 저하(Flickering/Lag)를 동반하고 코드가 불필요하게 복잡해집니다.
  * **대안 2: 드래그 이벤트 자체를 `useRef`로 추적하고 상단 컨테이너의 캡처(Capture) 페이즈에서 `click` 이벤트 가로채기 (채택)**
    - *장점*: React 컴포넌트의 추가적인 리렌더링 없이 드래그 여부를 Ref 상태로 안전하게 식별합니다. 브라우저의 이벤트 전파 흐름 중 타겟(Button)에 도달하기 전인 캡처 단계(`onClickCapture`)에서 드래그 변위가 5px을 초과한 경우 이벤트를 중단(`stopPropagation()`, `preventDefault()`)하여 자식 버튼의 `click` 핸들러 격발을 100% 깔끔하게 차단합니다.
    - *단점*: DOM 이벤트 흐름(Capture -> Target -> Bubble)에 대한 명확한 이해가 있어야 유지보수가 가능합니다.

* **최종 설계 합리화**:
  컴포넌트 리렌더링 최소화(성능 최적화)와 코드 구조의 단순함을 유지하기 위해 **대안 2(이벤트 캡처 단계 가로채기)**를 적용했습니다.
  1. `isDownRef`, `startXRef`, `dragMovedRef` 등의 `useRef`를 사용하여 마우스 무브 연산 시 리렌더링 없이 순수 DOM 스크롤값만 고정/조정했습니다.
  2. 드래그 이동 거리가 5px을 초과해 화면이 움직인 경우 `dragMovedRef`를 `true`로 설정하고, 컨테이너 엘리먼트의 `onClickCapture` 핸들러에서 이 플래그가 `true`이면 하위로 향하는 클릭 전파를 즉시 멈추도록 설계했습니다.
  3. 이에 추가적으로, 칩을 클릭 시 해당 칩이 컨테이너의 중앙에 자동으로 매끄럽게 정렬되도록 `scrollIntoView({ inline: 'center', behavior: 'smooth' })` API를 결합하여 Toss/배달의민족 수준의 세련되고 조작이 직관적인 가로 슬라이더 인터랙션을 완성했습니다.

### Q10. Next.js의 SSR(Server-Side Rendering) 및 빌드 환경을 고려하면서, 웹 애플리케이션 내에 플링코(Plinko) 스타일의 핀볼 물리 시뮬레이션을 Matter.js 같은 무거운 물리 라이브러리 없이 직접 구현하고 당첨 결과를 100% 신뢰할 수 있게 연동한 방법은 무엇인가요?

**A10. 문제 인식 ➔ 대안 분석 및 Trade-off 평가 ➔ 최종 설계 합리화**

* **문제 인식**:
  기존 원판 룰렛 대비 시각적 재미를 극대화하기 위해 핀볼 구슬이 핀들 사이를 튕겨 내려가 특정 식당 칸에 들어가는 게임(Plinko) 방식의 룰렛 모드를 추가하고자 했습니다. 이를 위해 2D 물리 연산이 필요한데, Next.js의 SSR 환경 특성상 `window`나 `document` 객체를 직접 참조하거나 WebAssembly 로더를 필요로 하는 기성 물리 엔진(예: Matter.js, Planck.js)은 빌드 실패, 초기 번들 오버헤드 증가(최소 100KB+), 그리고 Hydration mismatch 등 심각한 컴파일 및 런타임 제약을 수반합니다. 또한, 순수 물리 엔진은 랜덤 튕김으로 인해 최종 당첨 구슬이 백엔드/프론트엔드에서 pre-select한 당첨 식당 슬롯과 일치하지 않는 모순이 발생하여 당첨 무결성이 깨지게 됩니다.

* **대안 분석 및 Trade-off 평가**:
  * **대안 1: Matter.js 및 React-Matter-Wrapper 사용**
    - *장점*: 높은 품질의 리지드 바디(Rigid Body) 충돌 물리 및 바운스 연출 가능.
    - *단점*: Next.js build 시 WASM/Window 미정의 에러 우회용 dynamic import 처리 필요. 번들 크기가 매우 크고, 사전에 무작위로 선택한 당첨 결과 식당과 구슬이 굴러 떨어진 식당 슬롯을 매핑하려면 구슬 경로에 임의의 강제 외력을 복잡하게 보정해줘야 하므로 부자연스러운 움직임 발생 우려.
  * **대안 2: HTML5 Canvas 기반의 2D 경량 물리 엔진 직접 구현 (채택)**
    - *장점*: 약 150줄 남짓의 순수 수학 공식(벡터 반사, 중력가속도, 복원력 계수)만으로 작동하는 초경량(수 KB) 시뮬레이션 탑재 가능. SSR/Turbopack 환경과 100% 호환되며, 렌더링 오버헤드가 극도로 낮음. 당첨 구슬에만 특정 임계 높이(y=60) 이하부터 목적지(중앙 골드 슬롯)로 유도하는 미세한 횡방향 조향력(Steering Force)을 점진적으로 가함으로써, 부자연스러운 꺾임 없이 극도로 자연스럽게 목적지에 골인하도록 설계 가능.
    - *단점*: 다각형 충돌이나 정밀 마찰력 등 고난도 복합 물리는 표현하기 어려우나, 원형 구슬과 점(Peg) 간의 원호 충돌 및 벽 충돌만 해결하면 되는 핀볼 물리 특성상 대안 2로 완전 충족 가능.

* **최종 설계 합리화**:

  2. 무작위 구슬 분포로 인해 발생할 수 있는 교란을 해결하기 위해, 당첨 구슬에 대해서만 핀 영역을 통과한 후 중앙 당첨 슬롯을 향해 점진적 보정 드리프트(Corrective Drift)를 주어 당첨 결과 식당과 핀볼 골인 지점이 시각적으로 완벽하게 일치하는 고신뢰도 하이브리드 물리 엔진을 달성했습니다.
